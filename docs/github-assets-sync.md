# GitHub 资源同步到中国服务器方案说明

## 目标

这个文档只讨论方案，不进入编码实现。

目标是把 GitHub Releases 中的安装包或其他静态资源同步到中国境内的 Web 服务器，供安装器或下载页面使用。当前阶段先确定技术栈和安全模型，再决定是否开始开发。

## 需求边界

这个同步任务本质上比较简单，通常包含以下步骤：

1. 拉取 GitHub Release 或指定资源列表
2. 判断目标文件是否已经同步
3. 下载缺失文件或更新文件
4. 上传到中国境内服务器或对象存储
5. 记录同步结果，便于重试和审计

如果没有复杂工作流、没有高并发要求、也不需要大量业务逻辑，那么这个项目不应该选一个过重的技术栈。

## 当前规模判断

按目前已知情况：

- Syncthing 每个 release 大约有 34 个 assets
- 单个 release 总体积大约 500 MB
- 只保留最近 3 到 6 个版本
- 即使重复下载几次，整体成本也可接受

这说明当前任务规模并不大，更接近“小规模静态资源同步”，而不是需要复杂调度系统的分发平台。

基于这个规模，方案上可以更务实：

- 不需要为了节省少量重复下载而引入复杂状态系统
- 不需要先设计自定义上传协议
- 不需要一开始就上数据库或消息队列
- 优先考虑易部署、易排障、易重跑的实现

## 推荐的部署模型

如果这一侧只是负责从 GitHub 下载资源并放到本地目录，那么当前最值得优先考虑的模型其实是：

- 本地定时下载到目录
- 再通过 `rsync over SSH` 同步到中国服务器

也就是“下载阶段”和“发布阶段”分开：

1. 同步程序从 GitHub 拉取 Release 资源
2. 文件保存到本地一个稳定目录
3. 使用 `rsync` 把目录内容同步到中国服务器
4. 由中国服务器上的 Web 服务直接对外提供静态文件

这个模型比“自定义上传 API”更贴近你的实际需求，因为你的核心动作本来就是目录级文件同步，而不是复杂业务上传。

如果暂时不采用 `rsync`，次优才是“定时同步器 + 上传接口”这种简单模型：

- 一个定时任务程序负责轮询 GitHub Releases
- 发现新文件后下载到本地临时目录
- 再把文件上传到中国服务器
- 上传完成后记录元数据，例如版本、文件名、校验和、时间戳

这种模型的优点：

- 结构简单，容易排查问题
- 不依赖复杂消息队列
- 很适合少量静态资源同步
- 后续如果要迁移到对象存储，也比较容易改造

而 `rsync` 模型相对上传 API 还有几个额外优势：

- 天然适合目录同步
- 只传输差异内容，重复同步成本低
- 失败后重跑通常很直接
- 不需要额外设计上传协议
- 对静态资源发布非常实用

## 推送上传 vs 目标端主动拉取

这里其实有两种常见模式：

### 模式一：源端主动推送到中国服务器

流程：

1. 同步程序从 GitHub 下载文件
2. 同步程序把文件上传到中国服务器

优点：

- 同步链路更可控
- 可以在上传前做校验、重命名、过滤
- 目标服务器实现可以很薄，只需要接收上传
- 更容易做“同步成功才发布”的流程控制

缺点：

- 需要设计上传鉴权
- 需要处理上传失败重试
- 如果源端网络到中国方向较差，上传速度可能不稳定

### 模式二：目标端提供 HTTPS，由中国服务器主动拉取

流程：

1. 源端提供 HTTPS 文件列表或清单
2. 中国服务器定时去拉取清单
3. 中国服务器按需下载缺失文件

优点：

- 不需要自定义上传 API
- 不需要设计 HMAC 上传协议
- 目标端自己掌控下载、重试、落盘过程
- 对“国内服务器是最终控制面”的场景更自然

缺点：

- 需要源端稳定提供 HTTPS 文件服务或清单接口
- 如果源端本身不是静态文件服务，仍然要维护一个可访问出口
- 目标端逻辑会稍微重一些，因为它要负责比对、下载、校验

## 哪个更快

单看“速度”，没有绝对答案，主要取决于哪一段网络更快、更稳定。

可以把两种模式抽象成：

- 推送模式：`GitHub -> 源端 -> 中国服务器`
- 拉取模式：`GitHub -> 源端（生成清单）`，然后 `中国服务器 -> 源端` 或 `中国服务器 -> GitHub`

如果你的源端本身就在海外，并且中国服务器访问源端很稳定，那么“目标端主动拉取”通常更简单，也常常更快，因为：

- 中国服务器可以直接在本地控制并发和重试
- 不需要等源端上传完成后再确认
- 失败重试路径更短，排障更直接

如果你的源端已经把文件提前缓存在一个带宽较好的下载节点上，那么“目标端主动拉取”通常也更有优势，因为拉取过程完全由最终使用方所在区域控制。

但如果中国服务器的入站策略复杂、下载权限受限，或者你希望严格控制发布时机，那么“源端主动推送”更合适，虽然不一定更快，但更容易保证一致性。

### 实际判断标准

对这个项目，更应该优先看“稳定性和维护成本”，不是只看理论峰值速度。

在你的场景里，通常可以这样判断：

- 如果中国服务器可以稳定访问一个海外 HTTPS 源，就优先考虑“目标端主动拉取”
- 如果中国服务器不方便主动出网，或者你更想把流程集中在一个控制端，就选“源端主动推送”

## 当前更推荐哪一个

如果你的真实流程是“先把 GitHub 资源下载到一个目录，再把这个目录发布到中国服务器”，那我现在更偏向推荐 `rsync over SSH`。

原因：

- 它和你的工作流天然一致，都是目录级同步
- 不需要额外设计上传 API
- 不需要先设计 HMAC 协议
- 对重复执行很友好
- 对少量静态文件同步通常已经足够高效

更具体地说，这个场景下的推荐顺序我会调整为：

1. 第一选择：`rsync over SSH`
2. 第二选择：目标端主动拉取 HTTPS 清单或文件
3. 第三选择：自定义 HMAC 上传接口

如果只是同步 GitHub Releases 这类静态文件，并且文件量不大，`rsync` 往往是更务实的方案。

选择 `rsync` 的前提：

- 中国服务器可以通过 SSH 访问
- 你可以接受在服务器上维护 SSH key
- 发布方式本质上是目录同步，而不是对象存储 API

比较实用的一种落地方式是：

1. 本地程序定时从 GitHub 下载 Release 资源到一个 staging 目录
2. 下载完成后做一次校验，例如文件名、大小、哈希
3. 通过 `rsync` 同步到中国服务器的目标目录
4. 必要时在目标端做软链接切换或原子替换

这样做通常比“自定义 HMAC 上传接口”更省事，也通常比再专门设计一个 HTTPS 中转层更直接。

如果需要给下载页或安装器直接消费元数据，可以在同步阶段额外生成：

- `manifest.json`：资产级清单
- `releases.json`：版本索引
- `latest.json`：最新版本摘要

这样国内侧只需要读取静态 JSON 文件，不需要再做目录遍历或 GitHub API 调用。

## HMAC 签名上传是否足够安全

结论：对于这个场景，HMAC 签名请求通常是“够用的”，但前提是你把协议细节做完整，而不是只做一个很薄的签名参数。

如果满足以下条件，HMAC 可以认为是足够安全的：

- 全程使用 HTTPS
- 每个请求都带时间戳
- 每个请求都带随机 nonce，防止重放
- 签名内容覆盖 HTTP 方法、路径、查询参数、关键请求头、文件摘要
- 服务端校验时间窗口，例如 5 分钟以内
- 服务端对 nonce 做短期去重
- 密钥可轮换，并且按用途隔离
- 上传接口权限最小化，只允许必要路径和必要动作

如果缺少这些要素，HMAC 方案就会变得脆弱，尤其容易遇到下面这些问题：

- 请求被截获后重放
- 只签 URL，未签 body，导致内容可被替换
- 时间戳不校验，旧请求可重复利用
- 一把长期密钥权限过大，泄漏后影响面太大

所以，更准确的说法不是“HMAC 天然安全”，而是“设计正确的 HMAC 协议，在这个简单上传场景里通常足够安全”。

## HMAC 协议最少应包含什么

如果最终决定走 HMAC 上传，至少应包含这些字段：

- `timestamp`
- `nonce`
- `method`
- `path`
- `query` 的规范化结果
- 文件内容的 SHA-256 摘要
- `content-length`
- `content-type`
- `key-id`
- 最终签名值

服务端至少要校验：

- 时间窗口是否合法
- nonce 是否重复
- body 摘要是否匹配
- key-id 是否有效
- 签名是否正确
- 上传路径是否在允许范围内

## HMAC 与其他方案的对比

### 方案一：自定义 HMAC 上传接口

优点：

- 实现不复杂
- 不依赖第三方云厂商特性
- 可直接对接你自己的 Web 服务器
- 适合只有少量上传动作的内部工具

缺点：

- 需要自己设计协议细节
- 需要自己处理防重放和密钥轮换
- 需要自己做好审计和限流

适用场景：

- 你控制上传端和服务端
- 上传链路简单
- 文件量不大
- 希望系统依赖尽量少

### 方案二：对象存储预签名 URL

优点：

- 安全模型成熟
- 服务端逻辑更少
- 很适合文件上传

缺点：

- 依赖对象存储平台
- 如果最终目标是传统 Web 目录，需要再做一次同步或回源

适用场景：

- 最终文件可以直接放 COS、OSS、S3 兼容存储
- 希望减少自定义安全协议

### 方案三：SFTP/rsync over SSH

优点：

- 技术成熟
- 对简单文件同步很直接
- 不需要额外开发一套上传 API
- `rsync` 对目录增量同步效率高
- 非常适合“先下载到本地目录，再发布到远端目录”的流程

缺点：

- 审计和权限粒度往往不如专用上传接口细
- 在 Windows 环境或跨平台流程里体验可能一般
- 如果后续要做细粒度控制，扩展性较弱
- 如果目录结构管理不当，误同步风险需要靠发布流程约束

适用场景：

- 只有一台或少量服务器
- 运维上已经有 SSH 管理体系
- 希望最快落地
- 文件是静态资源
- 同步动作本质上是目录镜像或增量发布

## 语言选择建议

如果这是一个独立的小工具，而不是必须并入现有 Worker 代码库，那么语言选择可以按复杂度分层看。

在当前这个规模下，`bash` 已经是一个合理选择。

推荐 `bash` 的前提：

- 只是定时拉取 GitHub assets
- 下载后落到本地目录
- 用 `rsync` 发布到中国服务器
- 日志和状态需求比较轻

在这个前提下，`bash` 的优点是：

- 开发和部署成本最低
- 非常贴合 `cron + curl + jq + rsync` 这类流程
- 重跑简单
- 维护成本低

如果后面复杂度上升，再升级到 Go。

Go 仍然是更稳的长期选择，但不是当前阶段的必需品。

推荐 Go 的原因：

- 单个可执行文件，部署简单
- 做 HTTP 下载、上传、HMAC、SHA-256 都很直接
- 适合写成 cron job 或 systemd/supervisor 常驻任务
- 内存和依赖通常比 Node.js 更可控
- 后续如果要做并发下载或断点续传，也容易扩展

如果团队更熟悉 TypeScript，也可以选 Node.js/TypeScript，理由如下：

- 和当前项目语言一致
- 开发速度快
- HTTP 和 GitHub API 集成简单
- 对小型工具足够胜任

但要注意，Node.js 更适合“团队已经主要写 TypeScript”的情况，而不是因为这个任务本身天然更适合 Node。

Python 也能完成这个任务，但如果目标是长期运行、低依赖、易部署的同步器，它通常不是第一推荐。

## 当前建议

如果今天就要定一个偏稳妥、不过度设计的方案，我建议：

1. 如果目标是目录同步，优先选 `rsync over SSH`
2. 在当前规模下，同步器优先用 `bash`
3. 脚本依赖尽量控制在 `curl + jq + rsync + ssh + sha256sum`
4. 如果无法使用 SSH/rsync，再退到“目标端主动拉取”
5. 如果最终仍然决定走上传接口，鉴权再考虑 HMAC
6. 如果后面准备上对象存储，再考虑改成预签名 URL 模式

这个组合的原因很简单：

- 系统边界清楚
- 实现复杂度低
- 安全性在这个场景下足够
- 后续迁移空间也还可以

## 暂不建议做的事

在技术栈没有定下来之前，不建议先写下面这些内容：

- 具体上传 API 的字段设计
- 数据库存储结构
- 并发同步策略
- 自动重试和断点续传细节
- 部署脚本和 CI 流程

这些都依赖最终选择的是 Go、Node.js，还是直接走对象存储。

## 待确认问题

开始编码前，建议先确认这几个问题：

1. 目标端是传统 Web 服务器目录，还是对象存储
2. 同步程序运行在哪里，本地机器、GitHub Actions，还是中国服务器本机
3. 文件规模多大，是几十个文件还是长期累积数百上千个文件
4. 是否需要保留历史版本
5. 是否需要删除 GitHub 上已不存在的旧文件
6. 你更偏向 Go，还是更希望和当前仓库统一成 TypeScript
